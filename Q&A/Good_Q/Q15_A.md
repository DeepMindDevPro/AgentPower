## Skills 元数据格式与调用机制分析

### 一、Skills 元数据格式

Skills 元数据定义在 `SKILL.md` 文件的 YAML frontmatter 中，格式如下：

#### 字段说明：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `name` | string | ✅ | 技能名称，用于 `get_skill` 工具调用 |

### 二、Agent 如何决定调用 get_skill

整个流程基于 **Progressive Disclosure（渐进式披露）** 策略，分为三个层级：

#### **Level 1 - 元数据注入 System Prompt**

在 `SkillLoader.get_skills_metadata_prompt()` 方法中，生成仅包含 `name` 和 `description` 的元数据：

```python
def get_skills_metadata_prompt(self) -> str:
    """Generate prompt containing ONLY metadata (name + description)"""
    prompt_parts = ["## Available Skills\n"]
    prompt_parts.append("You have access to specialized skills.\n")
    prompt_parts.append("Load a skill's full content using the appropriate skill tool when needed.\n")
    
    for skill in self.loaded_skills.values():
        prompt_parts.append(f"- `{skill.name}`: {skill.description}")
    
    return "\n".join(prompt_parts)
```

**System Prompt 中注入的 Skills 元数据示例：**

## 设计模式的本质

这些模式的共同哲学是 **"Just-in-Time Disclosure"（即时披露）** 或 **"Progressive Disclosure"（渐进式披露）**：

| 特征 | 描述 |
|------|------|
| **减少认知负担** | 初始信息量控制在人脑可同时处理的范围内（7±2 原则） |
| **按需加载** | 只在需要时才展示详细信息，节省资源 |
| **主动选择** | 用户/客户端主动决定何时深入，保持控制感 |
| **性能优化** | 避免一次性加载大量数据导致的延迟 |
| **可扩展性** | 信息量增长时，初始界面不需要重新设计 |

## 为什么适合 LLM 场景

对于 LLM 来说，这种设计尤其重要：

1. **上下文窗口限制**：避免一次性注入几百 KB 的 Skill 内容
2. **token 成本**：只在需要时才消耗 token 加载详情
3. **注意力分散**：LLM 的注意力机制在信息过载时效率下降
4. **推理质量**：精简的元数据让 LLM 更容易做出正确的工具调用决策

## 🎓 **关键技术要点总结**

| 技术点 | 作用 |
|--------|------|
| **系统提示词注入技能元数据** | LLM 知道有哪些技能可用，以及它们的用途 |
| **渐进式披露（Progressive Disclosure）** | 只在需要时加载完整内容，节省 token |
| **Tool Schema 定义** | LLM 知道如何调用工具，需要传什么参数 |
| **语义匹配** | LLM 将用户问题的关键词与技能描述进行匹配 |
| **两层决策** | 1) LLM 决定是否调用 → 2) `GetSkillTool` 执行加载 |

---

## 💡 **为什么 LLM 能"理解"需要调用技能？**

1. **明确的指令**：系统提示词中写明了"**需要时使用 `get_skill` 工具获取完整指导**"
2. **技能描述作为上下文**：每个技能的 `description` 字段提供了语义线索
3. **工具调用能力**：LLM 通过工具 Schema 知道可以调用 `get_skill`，并传入 `skill_name`
4. **训练数据的影响**：现代 LLM 经过大量工具调用任务的训练，已经学会"根据问题类型选择工具"
