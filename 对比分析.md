Nanobot vs Mini-Agent 架构设计对比分析
一、项目定位差异
| 维度 | Nanobot | Mini-Agent |
|------|---------|-----------|
| 定位 | 面向生产的超轻量 AI Agent 平台 | MiniMax M2.5 模型的最佳实践演示项目 |
| 受众 | 需要长期运行、多渠道部署的用户 | 学习 Agent 构建、快速上手的开发者 |
| 核心承诺 | 可长期运行的个人 AI Agent，支持 14+ 聊天渠道 | 简洁可读的代码库，适合作为起点扩展 |
| 代码规模 | ~50 个 Python 模块，核心 Agent 循环 ~52KB | ~10 个 Python 模块，单文件 Agent ~22KB |

二、整体架构对比
Nanobot — 多层解耦架构：

渠道层(14+) → MessageBus(双队列) → AgentLoop → AgentRunner → LLMProvider(5种后端)
                                        ↑
              Session/Memory/Cron/Heartbeat/Command/Skills

Mini-Agent — 扁平直连架构：

CLI / ACP → Agent(直接持有LLM + Tools) → LLMClient → API
                    ↑
              Tools: File/Bash/Note/Skill/MCP

根本差异：Nanobot 通过 MessageBus 将渠道与 Agent 彻底解耦；Mini-Agent 的 Agent 直接持有 LLMClient 引用，调用链路更短，更易于阅读理解。

三、Agent 执行循环对比
Nanobot（agent/runner.py + agent/loop.py）
AgentLoop.run()          ← 产品层：会话锁、并发控制、命令路由
    └─ AgentRunner.run() ← 纯执行层：无产品逻辑
           ├─ ① 上下文治理（6步净化：孤立消息/token预算/微压缩）
           ├─ ② LLMProvider.chat_with_retry()
           ├─ ③ 工具执行（支持并发/独占/read_only分类）
           ├─ ④ 中途注入（最多5轮×3条）
           ├─ ⑤ 检查点发射（awaiting_tools/tools_completed）
           └─ ⑥ Hook生命周期（before_iteration/on_stream/after_iteration）

Mini-Agent（agent.py）
Agent.run()              ← 单类承担所有职责
    ├─ ① 强制截断（保留最近10条，防止溢出）
    ├─ ② _summarize_messages()（按用户轮次分段摘要）
    ├─ ③ LLMClient.generate()
    ├─ ④ 顺序执行 tool_calls（无并发）
    └─ ⑤ _check_cancelled()（asyncio.Event + 独立线程）

关键差异：

Nanobot 将 AgentLoop（产品层）和 AgentRunner（执行层）分离，AgentRunner 可被 SubagentManager 复用；Mini-Agent 合二为一，简洁但难以复用
Nanobot 有 6 步上下文治理管道，Mini-Agent 采用"强制保留最近10条"硬截断作为主要保护，摘要为辅
四、上下文/记忆管理对比
Nanobot — 三层记忆架构
Session(JSONL) → Consolidator(LLM摘要) → Dream(MEMORY.md + Git版本)
                         ↑                         ↑
                   token超预算时触发          每2小时定期蒸馏
                 AutoCompact(空闲压缩)

Consolidator：基于 tiktoken 精确计算，在 user-turn 边界截断，LLM 生成摘要（≤8000字符），最多 5 轮循环降低 token 至预算 50%
Dream：两阶段独立 Agent 循环，Phase1 分析 history.jsonl + git-blame 行年龄，Phase2 更新 MEMORY.md
GitStore（dulwich）：MEMORY.md/SOUL.md/USER.md 版本管理，支持 blame 行年龄注解
Mini-Agent — 轮次摘要策略
messages[] → _summarize_messages() → 按user轮次分段 → LLM摘要 → [Assistant Execution Summary]

触发条件：tiktoken 估算 > 25000 tokens 或 API 上报 total_tokens > 25000（双重检查）
摘要粒度：每个 user 消息之后的 assistant+tool 消息为一组，调用 LLM 压缩为 [Assistant Execution Summary]
无持久化记忆，跨会话信息通过 SessionNoteTool 写入 .agent_memory.json（JSON文件）
差异总结：Nanobot 是跨会话的三层持久记忆体系；Mini-Agent 是单次运行内的滚动压缩，跨会话记忆依赖工具手动调用。

五、LLM Provider 层对比
Nanobot — 注册表驱动，5种后端实现
# ProviderSpec 元数据声明（30+ 内置Provider）
@dataclass(frozen=True)
class ProviderSpec:
    name, keywords, backend, detect_by_key_prefix, detect_by_base_keyword, ...

# 后端实现
OpenAICompatProvider  ← 兼容大多数Provider（OpenRouter/DeepSeek/Qwen等）
AnthropicProvider     ← 原生 Anthropic SDK，支持 prompt caching/extended thinking
AzureOpenAIProvider
GitHubCopilotProvider ← OAuth 登录
OpenAICodexProvider   ← OAuth 登录

自动匹配：key 前缀 > base URL 关键词 > 模型名关键词 > gateway fallback
重试策略：standard（1/2/4s，3次）/ persistent（指数退避至60s，10次相同错误限制）
429 细分：余额不足 → 不重试；速率限制 → 重试 + Retry-After 头
Mini-Agent — 门面包装，2种实现
LLMClient(wrapper)
  ├─ AnthropicClient  ← anthropic SDK，支持 extended thinking blocks
  └─ OpenAIClient     ← httpx 直连，兼容 OpenAI 格式
  └─ OllamaClient     ← 本地 Ollama

路由逻辑：检测 api_base 域名是否为 MiniMax，自动拼接 /anthropic 或 /v1
重试：@async_retry 装饰器，指数退避（delay = initial × base^attempt），通用异常捕获
差异总结：Nanobot 的 Provider 系统是注册表元数据驱动，新增 Provider 零修改路由；Mini-Agent 是硬编码分支，Provider 数量少但每种实现更精细（Ollama、本地 Qwen 专项支持）。

六、工具系统对比
Nanobot
class Tool(ABC):
    read_only: bool      # 无副作用，可并发
    exclusive: bool      # 独占执行
    concurrency_safe     # 并发安全属性
    
    @tool_parameters({...})  # 装饰器冻结 schema（防篡改）
    cast_params()            # schema 驱动类型转换
    validate_params()        # JSON Schema 完整验证

ToolRegistry 定义缓存（builtin 字母序 + MCP 尾部），保证 prompt cache 命中率
工具错误追加 [Analyze the error above and try a different approach.] 提示
支持 read_only/exclusive 分类控制并发执行
Mini-Agent
class Tool(ABC):
    to_schema()         # Anthropic 格式
    to_openai_schema()  # OpenAI 格式（双格式支持）
    execute(**kwargs) → ToolResult(success, content, error)

ToolResult 明确区分 success/error，错误信息结构化
双格式 schema 输出（兼容 Anthropic 和 OpenAI tool_use 格式）
无 schema 冻结/类型转换，参数直接透传
工具功能对比：

| 功能 | Nanobot | Mini-Agent |
|------|---------|-----------|
| 文件读写 | ✅ ReadFile/WriteFile/EditFile/ListDir | ✅ ReadTool/WriteTool/EditTool |

| Web 搜索/抓取 | ✅ WebSearchTool + WebFetchTool（6种搜索引擎）| ❌（通过 MCP 集成）|
| 记忆持久化 | ✅ 自动（三层记忆架构） | ✅ SessionNoteTool（手动写 JSON）|
| 代码搜索 | ✅ GlobTool + GrepTool | ❌ |
| Notebook | ✅ NotebookEditTool | ❌ |
| 子 Agent | ✅ SpawnTool → SubagentManager | ❌ |
| 定时任务 | ✅ CronTool（三种调度类型） | ❌ |
| 后台进程 | ❌ | ✅ BashOutputTool（bash_id 管理）|
| 技能（Skills）| ✅ Markdown 注入，自动发现 | ✅ 渐进式披露（Level1/Level2）|
| MCP | ✅ 自动连接（stdio/SSE/HTTP，lazy） | ✅ 独立配置文件（mcp.json）|
| 自我工具（MyTool）| ✅ 运行时状态检查 | ❌ |

七、并发与取消机制对比
Nanobot
asyncio.Semaphore（默认3）全局并发限制
每 session_key 一把 asyncio.Lock 保证串行
活跃任务期间新消息进入 _pending_queues，工具执行间隙注入
/stop → 取消 asyncio.Task + SubagentManager 批量取消
_active_tasks[session_key] 跟踪每会话任务
Mini-Agent
asyncio.Event + 独立线程监听 Esc 键（不阻塞事件循环）
取消检查点：步骤开始前、工具执行前、每个工具执行后
取消后 _cleanup_incomplete_messages() 回滚未完成轮次
ACP 模式通过 cancel() 回调设置 state.cancelled 标志
八、接入层对比
Nanobot
CLI：prompt_toolkit，含 /stop//status//restart//history 等命令，onboard 交互式配置向导
14+ 聊天渠道：Telegram/Discord/WeChat/Feishu/DingTalk/Slack/Matrix/Email/QQ/WeCom 等
WebSocket 渠道：WebUI（React + Vite）
OpenAI-Compatible API：/v1/chat/completions + SSE，文件上传
Python SDK：Nanobot.from_config().run()
Mini-Agent
CLI：prompt_toolkit，支持 --task 非交互批量执行，内置 /help /clear /stats /log 命令
ACP（Agent Communication Protocol）：stdio 通信，集成 Zed Editor，每个会话独立 Agent 实例
无多渠道支持
九、配置系统对比
Nanobot
格式：JSON（~/.nanobot/config.json）
校验：Pydantic + pydantic-settings，支持 camelCase/snake_case 双写法（AliasChoices）
配置项：agents/channels/providers/api/gateway/tools 六大块，Provider 按名称索引
环境变量：自动读取（OPENAI_API_KEY 等）
Mini-Agent
格式：YAML（config.yaml）
校验：Pydantic BaseModel，手动 yaml.safe_load 解析
配置项：llm/agent/tools 三段，MCP 单独 mcp.json
搜索优先级：开发目录 > 用户目录 > 包目录（三级 fallback）
十、可观测性对比
Nanobot
Loguru：结构化日志
AgentHook：before_iteration/on_stream/after_iteration 生命周期可观测
进度推送：工具调用提示实时推送到渠道（send_progress/send_tool_hints）
Langfuse：可选接入（LANGFUSE_SECRET_KEY 环境变量激活）
流式输出：SSE/WebSocket 实时 delta
Mini-Agent
AgentLogger：每次 run 创建带时间戳的 .log 文件（~/.mini-agent/log/）
结构化 JSON 日志：记录完整 request/response/tool_result
终端彩色输出：步骤盒子、工具调用、结果展示，ANSI 颜色编码
/log 命令：打开日志目录（调用系统文件管理器）
十一、设计哲学总结
| 维度 | Nanobot | Mini-Agent |
|------|---------|-----------|
| 核心目标 | 生产可用的长期运行 Agent 平台 | 教学友好的最佳实践演示 |
| 扩展方式 | 注册表/插件/Hook 无侵入扩展 | 直接修改代码，示例驱动 |
| 记忆策略 | 三层自动持久记忆（短/中/长期）| 运行内压缩 + 手动工具写入 |
| 多模型支持 | 30+ Provider，自动路由 | 3种客户端，手动选择 |
| 多渠道 | 14+ 渠道，统一消息总线 | 仅 CLI + ACP（Zed Editor）|
| 并发设计 | session 级锁 + 全局 Semaphore | 单 asyncio 循环 + Esc 线程 |
| 代码可读性 | 模块化高，单文件职责单一 | 扁平直观，更易一遍读完 |
| 安全性 | SSRF 防护、workspace 沙盒、环境变量过滤 | 无专项安全设计 |
| 适合场景 | 部署为个人 Agent 服务器、多渠道机器人 | 学习 Agent 架构、快速 prototype |
