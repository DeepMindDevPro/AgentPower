在智能体落地的进程中，Day1~5 构建了骨架、神经、边界与手脚，**Day6 则要赋予它灵魂——即“如何思考”的内核程序**。

## 一、Prompt 工程：编写智能体的“大脑 BIOS”

如果把模型看作一个强大的通用 CPU，Prompt 就是运行其上的**最原始的操作系统**。它定义了角色、行为边界、思维流程和输出格式。好的 Prompt 不是“把需求说清楚”，而是**设计一个限定的认知环境**。

### 1. 系统级 Prompt 的四层结构
任何复杂智能体的系统提示，都可以解构为四个层：

- **角色层（Who）**：身份、专业领域、语气、底线。“你是一位严谨的金融分析师，拒绝任何猜测。”
- **能力与资源层（What）**：可用的工具、能访问的知识库。“你可以使用 `get_stock_data` 工具获取实时数据。”
- **行为协议层（How）**：思考的流程、工具调用规则、纠错机制。“在调用工具前，必须在 `<thinking>` 标签中写出计划……”
- **约束与格式层（Boundary）**：输出格式、安全限制、Token 预算。“最终回答必须用 JSON 格式输出，且 `summary` 字段不超过 100 字。”

**关键点**：行为协议层是智能体程序的核心，它演化为 Day2 的推理范式和今天要讲的思维链路。

### 2. 结构化输出引导
智能体的输出不仅是给人看的，更是给**代码解析器**和**下一轮推理**用的。必须在 Prompt 中强制定义：

- **JSON 模式**：直接给出 JSON Schema，并说明“只输出 JSON，不要有其他文字”。
- **语言标记封装**：如用 `<action>` `</action>` 包裹工具调用，用 `<final_answer>` 包裹最终回复。这比单纯靠“请输出JSON”更鲁棒，且易于解析。
- **模板化思考**：预留填写槽位，例如：“问题分析：___， 搜索关键词：___， 答案：___”。

这样做不仅能提高解析成功率，还能通过约束输出空间来**减少幻觉和跑偏**。

## 二、Thinking 标签：显式化思维流

`<thinking>` 标签（或类似的 `/* 思考 */`、`<thought>`）是 Prompt 工程中最强力的杠杆。它强制模型在执行任何外部动作（工具调用、最终回答）之前，**先进行一段内部独白**。

### 1. 为什么需要 Thinking 标签？
回想 Day2 的 ReAct 范式：“思考-行动-观察”。`<thinking>` 就是“思考”的容器。

- **状态保持**：在部分可观测环境中，Thinking 是智能体更新自身**信念状态**的地方。它总结已知信息，说出自己对当前局势的判断，相当于 Day1 记忆模块的**工作区快照**。
- **计划蓝图**：规划模块的每一次分解，都可以先输出一段思考：“目标：X。子任务：1.查A、2.算B、3.对比C。”这即是 CoT 的外化。
- **元认知监控**：反思模块需要自我审视。Thinking 里可以写：“刚才的工具返回错误，可能因为参数单位是公里而我给了米，我下次要换成公里重试。”这正是 Self-Refine/Reflexion 的精髓。
- **安全与可解释性**：让人或监控系统能看到智能体“为什么会这样做”，便于调试和拦截危险操作。

### 2. Thinking 标签的设计模式
不是简单加个 `<thinking>` 标记就有效，需要设计**内容骨架**。几种高价值模式：

**a) 分解式 Thinking（配 CoT/ToT）**
```
<thinking>
目标分解：
1. 理解问题：用户的深层需求是？
2. 检索信息：需要调用哪个工具？参数？
3. 方案评估：如果工具返回X，则答案可能是Y；如果返回Z...
4. 最终综合：
</thinking>
```

**b) 复盘式 Thinking（配 Self-Refine）**
```
<thinking>
上次行动回顾：我做了...，结果是...
偏差分析：预期是...，实际是...，原因可能是...
修正计划：接下来我将...
</thinking>
```

**c) 安全声明式 Thinking**
```
<thinking>
安全审查：本行动是否涉及PII？是否在权限范围内？
不确定性标注：我对...不确定，需要向用户确认。
</thinking>
```

### 3. 输出格式的“三段式”封装
典型的最佳实践是规范模型输出为：
```
<thinking>
内部推理过程（最终用户不可见或可选可见）
</thinking>
<action>
工具调用的 JSON
</action>
```
或最终回答时：
```
<thinking>
综合以上信息，答案应为...
</thinking>
<final_answer>
给用户的自然语言回复
</final_answer>
```
解析时，可以只提取 `<final_answer>` 展示给用户，但将 `<thinking>` 存入日志用于调试。这在工程上实现了**思考链与交付物的分离**。

## 三、思维链路优化实战指南

基于上述 Prompt 与 Thinking 结构，可以系统地优化智能体的“思维质量”。

### 1. 用提示词注入“思维检查清单”
在系统提示中直接写入自问语句，引导模型进行多角度审视：
> 在每次 `<thinking>` 中，你必须回答这三个问题：
> 1. 我是否已经掌握了回答此问题所需的全部事实？
> 2. 如果没掌握，我该用哪个工具获取？（不超过2个）
> 3. 我的推理是否有逻辑跳跃或未说明的假设？

这能将**反思能力先天植入每轮思考**，而非事后补救。

### 2. 上下文中的“思维摘要”防遗忘
长期对话中，在 `<thinking>` 结尾处要求输出一个**当前状态的总结**（50 tokens 以内），将其视为下一轮的高浓度记忆：
```
<state_summary>
用户目标：A；已完成：B；关键阻碍：C；下一动作：D
</state_summary>
```
这与 Day4 的自动摘要异曲同工，但在推理侧主动进行。

### 3. 错误恢复的标准化思维路径
不要依赖模型的随机应变，直接在 Prompt 中给出错误处理流程模板：
```
当工具返回错误时，你的思考必须按以下顺序：
1. 仔细阅读错误信息，提取错误代码和描述。
2. 判断错误类型：【参数错误/权限不足/服务不可用/未知】
3. 若是参数错误：分析错误提示中的正确格式，调整后最多重试1次。
4. 若是权限不足或服务不可用：向用户坦言相告，并提出替代方案。
5. 若是未知错误：记录原始返回，转为向人类求助。
```
这相当于把反思模块的某些逻辑固化为**硬编码的思维规则**，实现可控的韧性。

### 4. 思维链的剪枝与聚焦
Day3 提到 Token 是有限的。我们要在 Prompt 中告诫智能体控制思考长度：
> `<thinking>` 的内容不应超过 100 字，除非遇到必须深度分析的安全风险。请使用精炼的语言，只记录决策要点。

同时，优先使用**选择分支**而不是长长的单链推理。这有点手动 ToT 的意思：
```
<thinking>
方案A（直接搜索）：耗时短，可能信息不足。
方案B（逐步推断）：耗时长，较可靠。
鉴于用户要求速度，选A。
</thinking>
```

## 四、综合实战：一个高级智能体的 Prompt 骨架

结合 Day1 五组件、Day2 范式、Day5 工具，这里给出一个高度优化的 Prompt 设计片段，作为总览。

```markdown
你是 AutoResearcher，一个研究型智能助手。

## 配置文件
- 当前知识截止时间：2025-04
- 可用工具：`web_search`，`lookup_page`，`code_calculator`
- 安全策略：绝不执行文件删除命令，涉及隐私信息加密存储。

## 行为协议
你的所有回复必须严格遵循以下结构，且 **只有 `<final_answer>` 部分展示给用户**。

### 1. 当需要调用工具时：
<thinking>
- 用户意图：{一句话概括}
- 已知事实：{列出已掌握的关键信息}
- 缺失信息：{为回答还需要什么}
- 计划：{将调用哪个工具？参数为何？}
</thinking>
<action>
{{工具调用的JSON}}
</action>

### 2. 工具返回后，综合回答时：
<thinking>
- 已收到结果：{摘要}
- 逻辑推导：{基于事实得出结论的过程}
- 可信度：高/中/低，并说明理由
</thinking>
<final_answer>
{{给用户的完整回复，可使用Markdown}}
</final_answer>

### 3. 错误与异常处理
<thinking>
- 错误分析：{发生什么错误？可能原因？}
- 应对决策：{重试/更换工具/向用户澄清}
</thinking>
<!-- 然后输出新的action或final_answer -->

## 思维质量要求
- 每次思考必须包含对“是否值得调用工具”的判断：如果已有足够知识回答，直接输出 final_answer。
- 遇到数值数据，必须调用 code_calculator 计算，不得心算。
- 如果对搜索结果的权威性有怀疑，在 thinking 中明确标注。
```

这个骨架将**规划 (planning)、反思 (error handling)、记忆 (known facts)、工具使用 (action)** 整合进了一段紧凑的提示协议中。

## 五、回归基石：Prompt 是智能体程序的可编程体现

回到 AIMA 第 2 章的定义：**智能体程序 = 架构 + 程序**。在基于 LLM 的智能体中，**架构是运行环境与组件（Day1），程序的核心就是系统 Prompt 和思维链模板**。你通过 Prompt 规定了智能体的函数：如何把感知序列映射到行动。而 `<thinking>` 标签，就是在该映射中插入递归的自省和规划步骤。

Day2 的推理范式是**宏观策略**，而 Prompt 和 Thinking 标签就是**微观实现**。你通过编写 Prompt，决定了智能体会采用 ReAct、CoT 还是 ToT 的思维风格，并控制其反思的深度与频率。这是人与智能体之间最高效的控制接口。
