# Agent 底层通用原理手册

> 基于《人工智能：一种现代方法》第2章与第17章理论基石，整合现代自主智能体工程的七个核心维度。本手册既是原理纲要，也是痛点解决速查表。

---

## 卷首：基石理论（AIMA 第2章 & 第17章）

### 智能体的定义
**智能体** = 架构 + 程序。任何通过**传感器**感知环境、通过**执行器**作用于环境的事物。理性智能体选择**期望效用最大化**的行动，取决于性能度量、先验知识、可行动作与感知序列。

### PEAS 任务描述标准
| 组件 | 含义 | 示例（自动驾驶） |
|------|------|------------------|
| P 性能度量 | 定义成功 | 安全行驶里程、乘客舒适度 |
| E 环境 | 所处世界 | 道路、其他车辆、行人 |
| A 执行器 | 能做什么 | 方向盘、油门、刹车、指示灯 |
| S 传感器 | 能感知什么 | 摄像头、激光雷达、GPS |

### 环境性质的分类轴
| 维度 | 类型 | 设计影响 |
|------|------|----------|
| 可观测性 | 完全/部分可观测 | 需否维护内部状态 |
| 智能体数量 | 单/多智能体 | 需否博弈论建模 |
| 确定性 | 确定/随机 | 需否概率推理 |
| 交互模式 | 片段式/序贯式 | 需否长程规划 |
| 变化速度 | 静态/动态 | 需否实时响应 |
| 状态空间 | 离散/连续 | 需离散化还是连续优化 |

### 多智能体核心概念
- **纳什均衡**：每个玩家都对他人策略做出了最优反应，无人愿单方面偏离。
- **占优策略**：不管别人怎么做，这个策略总是更优。
- **子博弈完美均衡**：剔除了空洞威胁的真正可信策略。
- **机制设计**：设计规则使自利玩家在均衡中实现设计者想要的社会结果。

## 第一章：Agent 核心五组件架构

### 1.1 五大组件定义

| 组件 | 理论来源 | 职责 | 类比 |
|------|---------|------|------|
| **规划** | 基于目标/效用的智能体，序贯决策 | 将高层目标分解为子任务序列 | 大脑前额叶（决策与计划） |
| **记忆** | 内部状态，感知序列历史 | 存储、检索、更新信息 | 海马体+工作记忆 |
| **工具** | 执行器的能力扩展 | 操作外部世界，获取实时信息 | 手与感官延伸 |
| **反思** | 性能度量评估，效用优化 | 自我评估、纠错、优化 | 元认知 |
| **执行** | 行动输出 | 按计划执行具体动作，收集反馈 | 运动皮层 |

### 1.2 通用架构图
```
┌─────────────────────────────────────────────────────────┐
│                    🌐 环境 / 用户                        │
└─┬──────────────────────────────────────────────┬────────┘
  │ 任务指令                          结果 / 行动反馈
  ↓                                              ↑
┌─┴─────────────── 智能体核心 ───────────────────┴────────┐
│                                                          │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐  │
│  │   规  划     │←──→│   记  忆     │←──→│   反  思     │  │
│  │  Planning   │    │  Memory     │    │ Reflection  │  │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘  │
│         │ 生成计划         │ 提供上下文        │ 产生反馈  │
│         ↓                  ↓                  ↓          │
│  ┌──────────────────────────────────────────────┐        │
│  │              执行 (Execution)               │        │
│  └──────────────────┬───────────────────────────┘        │
│                     │ 调用工具                          │
│                     ↓                                   │
│  ┌─────────────────────────────────────┐                │
│  │          工 具 集 (Tools)            │                │
│  └─────────────────────────────────────┘                │
└────────────────── 执行结果返回记忆，反思介入 ──→ 循环
```

**交互流程**：
1. 任务进入记忆工作区。
2. 规划模块读取目标与历史，生成行动计划，写回记忆。
3. 执行模块按计划调用工具，结果存入记忆。
4. 反思模块评估执行结果，提出修正建议，更新计划。
5. 重复 2-4，直至目标达成。

## 第二章：推理范式图谱

智能体的“思维风格”由这四种范式及其组合定义。

| 范式              | 思维结构 | 外部交互 | 核心机制 | 典型场景 | 失败模式 |
|-----------------|---------|----------|----------|----------|---------|
| **ReAct**       | 推理-行动循环 | 强（调用工具） | 一步一步与外部交换信息 | 事实核查、多跳问答、网页导航 | 无效循环、行动冗余 |
| **CoT**         | 线性链条 | 无 | 多步逻辑展开 | 数学推理、代码逻辑 | 一步错步步错 |
| **ToT**         | 树状搜索 | 无（或内部评分） | 生成多条路径并评估剪枝 | 创意写作、策略规划 | 搜索空间爆炸、评估不准 |
| **Self-Refine** | 迭代螺旋 | 可有外部评价 | 对自身输出批评后改进 | 文本润色、代码优化 | 过拟合偏好、循环无提升 |

### 场景速查决策树
```
需要外部实时信息？ ──是→ ReAct
否 ↓
可以线性推导？ ───是→ CoT
否 ↓
需要多方案比较或创意？ ──是→ ToT
↓
已有初稿要优化？ ──────是→ Self-Refine
```
**组合增强**：ReAct + CoT（每次推理深度分析），ToT + Self-Refine（叶节点打磨），ReAct + Self-Refine（全局轨迹自评）。

## 第三章：Token 工程与上下文管理

### 3.1 Token 本质与估算

- **Token** 是模型理解的文本最小单元，不等于单词或字符。
- 主流分词法：字节对编码（BPE），中文单字通常被拆为字节序列再组合，**1 个中文汉字 ≈ 1.6–2 个 Token**（因模型而异）。
- **估算工具**：`tiktoken`（OpenAI）、`transformers.AutoTokenizer`、在线分词可视化。
- **实战心法**：必须用目标模型的分词器**精确计算**，拒绝通用估算公式。

### 3.2 上下文窗口原理

- 上下文窗口是训练时确定的硬性限制，由位置编码与注意力矩阵尺寸决定。
- 每次请求需将系统提示、对话历史、当前消息全部打包输入，**总 Token 数不能超过窗口**。
- 注意力计算复杂度 \(O(n^2)\)，位置编码有“中间丢失”（Lost in the Middle）现象——窗口中间的信息检索率最低。

### 3.3 超限原因

| 原因 | 机制 | 后果 |
|------|------|------|
| 历史堆叠 | 每轮对话都累积进输入 | 最早信息被截断或遗忘 |
| 推理膨胀 | CoT/ToT 消耗数千 Token 思考 | 迅速吞噬窗口 |
| ReAct 循环 | 多次思考-行动-观测都记录 | 指数级占用 |
| Self-Refine 迭代 | 多版本生成 | 冗余历史 |

### 3.4 四大工程解决方案

| 策略              | 原理 | 优点 | 缺陷 | 适用 |
|------------------|------|------|------|------|
| **固定轮数截断**    | 只保留最近 N 轮 | 极简，稳定 | 机械失忆 | 短期任务，兜底 |
| **动态截断**       | 按重要性或相关性保留历史 | 保留关键信息 | 实现复杂，需要向量库 | 长程专业对话 |
| **自动摘要**       | 用模型将历史压缩为摘要 | 压缩率高，维持连贯 | 细节丢失，摘要幻觉风险 | 长程任务追踪 |
| **缓存机制**       | 复用 KV Cache、语义缓存 | 降低延迟与成本 | 需要推理框架支持 | 高吞吐场景 |

### 3.5 协作架构
```
新消息 → 写入长期日志 → 检查工作记忆Token占用 →
  未超限 → 直接请求
  超限 → 动态截断（检索重要历史块 + 最近M轮）
        → 仍超限 → 自动摘要压缩
        → 最后固定截断兜底 → 语义缓存检查 → 请求
```

## 第四章：插件体系与工具标准化

### 4.1 Skill 的本质
**Skill = 意图识别 + 交互范式 + 工具编排 + 知识上下文**
是工具之上的任务导向封装，让智能体理解“何时用、怎么用、用完该怎么办”，而非裸函数。

### 4.2 MCP 协议（Model Context Protocol）

**架构**：
```
Host (AI应用) → MCP Client → JSON-RPC over stdio/HTTP → MCP Server (工具/资源/提示)
```

**三种原语**：

| 原语 | 用途 | 模型触发方式 |
|------|------|-------------|
| **Tools** | 可执行的函数调用 | 模型输出 tool_call |
| **Resources** | 静态/动态数据读取 | read_resource 请求 |
| **Prompts** | 预定义对话模板 | 用户选择，注入上下文 |

**传输**：`stdio`（本地，零网络配置）或 `Streamable HTTP`（远程）。

**MCP 与 Function Calling 关系**：前者是标准化、分布式工具发现与调用协议；后者是模型输出调用的具体能力。MCP 负责“有哪些工具”，Function Calling 负责“此时该调谁”。

### 4.3 Skill 与 MCP 的差异

| 维度 | Skill | MCP |
|------|-------|-----|
| 抽象层次 | 任务导向，含业务逻辑 | 通信标准，提供原子能力 |
| 标准化 | 平台特定 | 开放协议，跨平台互操作 |
| 类比 | 电子邮件App | SMTP/POP3协议 |

**协同**：MCP 构建标准工具连接，Skill 封装多个 MCP 调用完成完整任务。

### 4.4 开发规范

1. **描述即界面**：精确、枚举限制、Schema 严格。
2. **错误善意化**：返回结构化错误，含修正建议。
3. **安全最小权限**：凭据隔离、输入二次校验、副作用声明。
4. **幂等与超时**：读操作幂等，写操作设计幂等键；超时后返回可重试错误。
5. **测试策略**：单元测试 → 集成测试 → Model-in-the-loop 测试（关键验收）。

## 第五章：Prompt 工程与思维链路优化

### 5.1 系统 Prompt 的四层结构
| 层级 | 定义 | 示例 |
|------|------|------|
| 角色层 | 身份、语气、底线 | “严谨的金融分析师，拒绝猜测” |
| 能力资源层 | 可用工具、知识库 | “可调用 get_stock_data” |
| 行为协议层 | 思考流程、纠错机制 | “调用前必须在 thinking 中写计划” |
| 约束格式层 | 输出格式、Token 预算 | “最终用 JSON 输出” |

### 5.2 Thinking 标签的四大模式

**a) 分解式（配 CoT/ToT）**
```
<thinking>目标分解：1.理解问题 2.检索信息 3.方案评估 4.综合</thinking>
```

**b) 复盘式（配 Self-Refine）**
```
<thinking>上次行动回顾... 偏差分析... 修正计划...</thinking>
```

**c) 安全声明式**
```
<thinking>安全审查：是否涉及PII？不确定处标注。</thinking>
```

**d) 状态摘要式（防遗忘）**
```
<state_summary>用户目标：A；已完成：B；关键阻碍：C；下一动作：D</state_summary>
```

### 5.3 输出格式三段式封装
```
<thinking>内部推理</thinking>
<action>工具调用JSON</action>

// 或最终回答时：
<thinking>综合推理</thinking>
<final_answer>给用户的文本</final_answer>
```
解析时只提取 `<final_answer>` 展示，`<thinking>` 存档调试。

### 5.4 思维质量优化清单

- **思维检查清单**：在 Prompt 中注入自问语句（事实是否充足？需要哪个工具？逻辑有跳跃吗？）。
- **错误恢复标准路径**：按错误类型（参数/权限/未知）定义固定处理模板，避免随机应变。
- **思维长度控制**：限制 thinking 字数，优先使用选择分支而非长推导，手动实现轻量 ToT。

## 附录：工程痛点速查表

| 痛点 | 病因 | 解决方案 | 涉及章节 |
|------|------|---------|---------|
| 多轮后突然“失忆” | 历史堆叠超出上下文窗口 | 固定截断兜底+动态检索+自动摘要 | 第三章 |
| 长对话逻辑断裂 | 中间截断丢失关键信息 | 按重要性动态截断，结构化摘要 | 第三、四章 |
| 工具调用死循环 | 错误信息不友好，模型盲目重试 | 返回结构化错误+建议，限制最大重试次数 | 第四章 |
| 推理成本过高 | CoT 过长，ToT 搜索过宽 | 限制 thinking 长度，早剪枝，语义缓存 | 第二、三章 |
| 工具选择错误 | 描述模糊或 Schema 不严格 | 精确描述+JSON Schema+枚举限制 | 第四章 |
| 思考与交付混杂 | 未分离内部推理与用户可见输出 | 三段式标签封装，解析层过滤 | 第五章 |
| 多智能体决策冲突 | 纳什均衡认知不足 | 博弈论建模、引入机制设计 | 卷首 |
| 非平稳环境收敛失败 | 其他智能体在学，环境在漂移 | 理解多智能体RL的非平稳性，设计稳健策略 | 卷首 |
| 环境分类错误导致选错范型 | 未用 PEAS 做任务分析 | 先套用 PEAS 模板，再选择架构与范式 | 卷首 |
