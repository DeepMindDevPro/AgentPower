在 Day3 我们诊断出了超长对话超限的病根：上下文窗口硬性限制、注意力复杂度、历史堆叠与推理膨胀。
今天则要开出具体的工程药方——**四种关键的上下文管理策略**。它们是智能体 Memory 模块的“新陈代谢系统”，让智能体在有限窗口内持续保持清醒与效率。

---

## 一、固定轮数截断 (Fixed-Round Truncation)

### 方案原理
只保留**最近 N 轮完整对话**，更早的历史直接删除。N 是一个经验值，根据窗口大小和平均单轮消耗 Token 数倒推。
- 计算方式：`N = floor( (窗口大小 - 系统提示Token - 当前输入预留) / 单轮平均Token )`
- 实现：对话历史用数组存储，每次追加新消息后，如果长度超过 `2N` 条消息（每轮包含 user + assistant），则从头部丢弃直至满足长度。

### 优点
- 实现极简，无额外推理开销。
- 保证延迟稳定，不会因历史增长而变慢。

### 缺陷
- **机械性失忆**：重要的早期指令、关键事实、角色设定可能因轮数靠前而被直接丢掉。
- 对信息密度不均匀的对话适应性差（某轮的重要决策信息与寒暄同等对待）。

### 适用场景
- 对话模式简单、不依赖早期设定的应用（如单次任务型问答、短期客服）。
- 作为其他高级策略的**兜底安全阀**，防止窗口溢出报错。

## 二、动态截断 (Dynamic Truncation)

### 方案原理
不再以轮数为单位，而是**根据消息的重要性或相关性动态选择保留/丢弃**。核心包括：
1. **基于重要性的剪枝**：对每条历史消息进行打分（可通过小型分类模型、规则或 LLM 自评），丢弃得分最低的消息，直到总 Token 数降至安全范围。
2. **基于相关性的检索**：将历史消息向量化存储，每次需要时，用当前用户查询作为 query，检索出最相关的 K 条历史消息，与最近几轮完整上下文拼接。这是**RAG 的变体**，相当于把长期记忆检索引入工作记忆。
3. **混合策略**：保留最近 `M` 轮完整对话（保证局部连贯），更早的消息通过检索注入。

### 优点
- **最大程度保留关键信息**，避免机械截断的“一刀切”。
- 鲁棒性强，既能保留早期角色设定，又能引入远期但相关的知识。

### 缺陷
- 实现复杂度较高，需要向量数据库或评分模型支持。
- 如果检索不准，可能引入无关信息，反而干扰推理。

### 实现要点
- 分块与向量化：将对话按轮次或语义块切分，生成嵌入存入向量库。
- 检索窗口拼接：`[System Prompt] + [检索出的历史块] + [最近M轮对话] + [当前询问]`。
- 注意格式：检索出的历史块需重新拼装成标准的对话格式，保证模型可理解。

### 适用场景
- 长程专业对话、角色扮演、需反复引用先前决策的复杂任务。

## 三、自动摘要 (Automatic Summarization)

### 方案原理
用模型自身（或一个更轻量模型）将长段历史**压缩为摘要**，用摘要替代原始记录占据上下文空间。相当于**知识蒸馏**，保留语义精华，丢弃冗余细节。

### 分层摘要策略
- **逐步摘要**：当对话达到一定长度（如 20 轮），触发一次摘要调用，将前 N 轮压缩为一段摘要文本，插入到历史中替代原对话。
- **增量摘要**：新消息到达时，将现有摘要与新的一批消息合并，生成更新后的摘要。数学上类似在线递归：  
  `摘要_new = 摘要_old + 新消息队列 → 模型生成浓缩版`
- **结构化摘要**：按主题或任务拆分为多段摘要，如：
  - **角色设定摘要**：用户偏好、角色要求（长期有效，极少更新）。
  - **任务进度摘要**：已完成步骤、当前状态、未完成事项。
  - **关键事实摘要**：对话中提及的时间、地点、人名、约束条件。

### 优点
- 压缩率极高（10:1 甚至更高），能极大延长有效对话轮数。
- 维持连贯感，模型看到的是“用户想要什么、我们做到了哪里”，而非支离破碎的原始对话。

### 缺陷
- **信息丢失风险**：摘要必然会丢失细节，对于需要精确引用原文的场景不适用。
- 摘要质量依赖模型能力，可能产生扭曲或幻觉。
- 额外推理成本与延迟增加。

### 适用场景
- 长程、任务导向型对话（如虚拟助理、长期项目管理）。
- 需要持续追踪全局状态，而非精确回溯每句话的场景。

## 四、缓存机制 (Caching)

### 方案原理
利用 Transformer 推理过程中的 **KV Cache（键值缓存）** 性质，避免重复计算相同前缀的注意力结果。
- 在推理时，对已经计算过的前缀 Token 序列，保存其每层的 Key 和 Value 张量。
- 当下一轮输入包含此前缀时，直接复用保存的 KV 张量，只计算新增部分的注意力。
- **直接效益**：显著降低首 Token 延迟和计算成本，但不减少 Token 占用。

### 更广义的缓存
- **提示缓存（Prompt Cache）**：将系统提示和静态指令预先计算并缓存，极大加速大量并发请求的冷启动。
- **语义缓存（Semantic Cache）**：将用户查询的嵌入向量与已缓存响应的查询嵌入进行相似度匹配，如果命中且缓存未过期，直接返回存储的响应，完全绕过 LLM 生成。适用于高频固定问答。
- **工具结果缓存**：对函数调用结果设置 TTL 缓存，避免重复的工具调用开销。

### 优点
- KV Cache 对用户透明，可无缝加速多轮对话，尤其是系统提示很长的场景。
- 语义缓存能成倍降低 API 费用和延迟。

### 缺陷
- KV Cache 需要后端推理框架支持（vLLM、TensorRT-LLM 等），且对并发管理要求高（内存占用极大）。
- 语义缓存需要专用的缓存管理策略，防止过期数据误导。

### 适用场景
- 任何需要高吞吐、低延迟的对话系统。
- 系统提示固定、多用户共享前缀的 API 服务。

## 四策联用：架构落地

这四种机制并非互斥，而是分层协作，构成智能体 Memory 模块的**生命周期管理**：

| 机制 | 在架构中的位置 | 作用 | 示例触发条件 |
|------|---------------|------|-------------|
| **固定轮数截断** | 工作记忆的最后一道防线 | 兜底防止窗口溢出 | token 数 > 安全阈值的 95% |
| **动态截断** | 工作记忆与长期记忆的桥梁 | 按需检索重要历史 | 用户问题涉及过去信息时 |
| **自动摘要** | 记忆的压缩器/归档器 | 将冗余历史蒸馏为状态摘要 | 每 10 轮对话无异常时触发 |
| **缓存** | 推理引擎层 + 工具层 | 加速重复计算与响应 | 每次 API 请求前检查 |

**标准流程示例**：
1. 新消息到达。
2. 将消息写入完整历史日志（长期存储）。
3. 检查当前工作记忆（上下文窗口）Token 占用：
   - 若未超限，直接构造输入并请求。
   - 若超限，启动**动态截断**：用当前消息检索长期记忆，取出高相关历史块，并与最近 M 轮拼接。
4. 在动态截断后仍可能溢出时，对早期部分执行**自动摘要**，用摘要替代原始文本。
5. 最后**固定截断**确保绝对不超过模型上下文限制。
6. 请求时，检查**语义缓存**：若当前查询与缓存的某个查询高度相似，直接返回缓存响应。
7. 生成响应后，更新 KV Cache（基础设施自动完成）。

通过这种分层、多级防御的设计，Token 限制从“致命瓶颈”变成了“可优雅管理的资源”。

- **反思模块**的评估结果可标记出哪些历史消息是关键决策点，赋予这些消息更高的重要性权重，使其免于被截断/摘要丢弃。
