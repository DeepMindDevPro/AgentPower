## 一、Skill 的本质：可复用能力单元

在五组件架构中，“Skill”不是原子化的单个 API，而是**将规划、记忆、反思、执行封装在一起的、面向特定任务的能力胶囊**。

- **定义**：Skill = 意图识别 + 交互范式 + 工具编排 + 知识上下文。
- **形态**：
  - **轻量级**：带描述的单个函数，如 `get_weather(city: str) -> dict`。
  - **重量级**：一个微智能体，内部可能包含自己的规划与反思，如“撰写并发送周报”Skill。
- **核心构成**：
  - **接口签名**：输入参数、输出结构、异常定义。
  - **元描述**：自然语言的功能说明、适用场景、使用限制，供主智能体的**规划模块**决策是否调用。
  - **内部逻辑**：可以是纯代码、模型推理、其他工具组合甚至多步 ReAct 循环。
- **在设计层面，Skill 是“工具”的语义化封装**：工具只提供裸函数，Skill 提供“何时用、怎么用、用完之后该怎么办”的上下文智慧。

## 二、MCP 协议完整解析：连接工具的“USB-C”

若每个 AI 应用都对接着成百上千个不同的工具 API，碎片化无法避免。**MCP（Model Context Protocol）** 正是为解决此 M×N 集成问题而生的开放标准，它定义了模型与外部工具、资源之间标准化的通信协议。

### 1. 架构角色
MCP 采用客户端-服务器架构，包含三个角色：
- **MCP Host**：AI 应用本身（如 Claude Desktop、VS Code、自定义智能体）。它包含执行决策的模型运行环境。
- **MCP Client**：运行在 Host 内部的协议客户端，与 Server 维持一对一连接，负责翻译 Host 的指令为协议请求。
- **MCP Server**：轻量级服务，暴露特定能力：工具、资源、提示模板。每个 Server 专注做一件事。

```
┌─────────────── Host (Agent) ───────────────┐
│  ┌──── 规划器 / ReAct循环 ────┐            │
│  │ 决策：需要查询天气         │            │
│  └─────────────┬──────────────┘            │
│                │ 调用工具                  │
│  ┌─────────────▼──────────────┐            │
│  │        MCP Client          │            │
│  └─────────────┬──────────────┘            │
└────────────────┼───────────────────────────┘
                 │ JSON-RPC over stdio/HTTP
┌────────────────▼───────────────────────────┐
│          MCP Server (天气服务)             │
│  tools: { "get_weather": {...} }          │
│  resources: { "weather://cities": [...] } │
└───────────────────────────────────────────┘
```

### 2. 核心原语（Primitives）
MCP Server 可以暴露三类能力，由 Client 发现并调用：

| 原语 | 用途 | 模型如何触发 | 举个栗子 |
|------|------|-------------|--------|
| **Tools** | 模型可执行的函数调用 | 模型输出 tool_call 指令 | `get_weather(city="北京")` → 返回 JSON |
| **Resources** | 应用可读取的静态/动态数据 | 模型通过 `read_resource` 请求访问 | `project://files/main.py` 获取代码 |
| **Prompts** | 预定义的对话模板 | 用户选择后注入上下文 | “帮我审查这段代码”模板 |

**Tools** 是动态交互的核心：Server 可以定义多个工具，每个工具有 JSON Schema 描述。Client 在初始化时获取工具列表，将其转化为模型认识的 function definitions，供模型决策。模型调用后，Client 将调用请求转发给 Server 执行，并将结果返回模型。这是智能体“工具”模块的标准实现路径。

### 3. 传输与生命周期
- **传输协议**：
  - `stdio`：Client 作为子进程启动 Server，通过标准输入输出传输 JSON-RPC 消息。**最常用、零网络配置、适用于本地工具**。
  - `Streamable HTTP`：（演进自 SSE）基于 HTTP 的双向流式通信，适用于远程或 Web 部署。
- **生命周期**：
  1. **初始化**：Client 发送 `initialize` 请求，Server 返回能力清单（支持的原语、版本等）。
  2. **正常运行**：工具发现/调用/通知，持续交互。
  3. **关闭**：连接关闭，Server 清理资源。
- **安全性**：协议强调用户确认机制（如工具调用前需用户批准），并建议 Server 做输入校验、沙箱隔离。权限控制由 Host 层实现。

### 4. MCP 与 Function Calling 的关系
- **Function Calling** 是模型能力：模型输出 `tool_calls` 字段，表示想调用某个函数。
- **MCP** 是分布式工具标准：规定了如何发现、描述、安全调用这些函数，且将工具实现与模型应用解耦。
- **它们是互补的**：一个智能体应用可以用 MCP 连接十个 Server，每个 Server 暴露若干函数；模型通过 Function Calling 机制选择调用哪个，由 MCP Client 路由过去。

## 三、Skill 与 MCP 的差异

许多框架将可复用能力称为 Skill（Coze 的插件、Dify 的工具、Semantic Kernel 的 Skill/Plugin），它们与 MCP 的区别在于抽象层次和应用范式。

| 维度 | Skill（平台/框架级） | MCP（协议级） |
|------|-------------------|-------------|
| **抽象高度** | **任务导向**：一个Skill可以完成“发邮件”这个完整任务，内含鉴权、模板、发送、检查。 | **通信标准**：定义如何发现工具、如何调用，不关心业务逻辑。 |
| **组合性** | 一般由平台内部进行组合，可能有自己的编排引擎。 | 纯粹提供原子能力，组合由 Host 的规划器完成。 |
| **标准化** | 平台特定，一个 Coze 插件不能在 Dify 直接用。 | 开放协议，任何实现了 MCP 的组件都可以互操作。 |
| **实现依赖** | 可能直接包含模型推理、自有状态机。 | 仅实现协议的轻量服务，通常不包含 AI 逻辑。 |
| **类比** | “电子邮件App” | “SMTP/POP3协议” |

**协同关系**：越来越多的框架将 MCP 作为底层标准。你可以用 MCP 来构建一个“数据库查询工具服务器”，然后将其封装成一个“数据分析 Skill”，在 Skill 内部组合多个 MCP 调用。**Skill 定义 What 和 When，MCP 提供 How 的标准化通道。**

## 四、插件/Skill 开发规范

不论是为某个平台开发 Skill，还是写一个 MCP Server，良好规范决定了智能体用起来是否“顺手”。

### 1. 描述即界面
模型的唯一“说明书”是你的文字描述与参数 Schema，务必：
- **精确**：`get_weather` 的描述应写明“获取指定城市的实时天气，返回温度、湿度、天气状况”。
- **注明限制**：“仅支持省会城市”，“需要网络连接”。
- **Schema 严格**：参数类型（string/number）、枚举值、默认值、必选/可选清楚定义。使用 JSON Schema 书写。

### 2. 错误善意化
当工具执行失败，返回的错误信息需让**模型能理解并修正**：
```json
{
  "error": "CityNotFound",
  "message": "未找到城市 '北惊'，您是否想指 '北京'？",
  "suggestions": ["北京", "南京"]
}
```
同时提供可用于重试的详细信息，避免模型无意义死循环。

### 3. 安全与最小权限
- **输入校验**：工具内部必须二次校验所有参数，防止注入或异常。
- **敏感信息隔离**：凭据/API Key 禁止暴露给模型，由 Server 侧通过环境变量注入。
- **资源限制**：单次返回数据量有上限（例如前 50 行），避免撑爆上下文窗口。提供分页参数。
- **副作用声明**：如果工具会修改数据（写操作），必须在描述中明确，并要求用户确认（MCP 层面可开启）。

### 4. 幂等性与超时
- **读操作**应保持幂等，多次调用返回相同结果。
- **写操作**尽量设计为幂等（如使用唯一请求 ID），防止因网络重试导致重复执行。
- 设置合理超时（如 30 秒），超时后返回结构化错误，模型可以决定是否重试。

### 5. 测试策略
- **单元测试**：纯逻辑部分。
- **集成测试**：在模拟 MCP 客户端下调用，检查 JSON-RPC 交互。
- **Model-in-the-loop 测试**：让真实模型使用你的工具，观察其决策质量和错误恢复能力。这是最关键的验收测试。

### 6. MCP Server 开发快速上手
官方提供 TypeScript / Python SDK。以 Python 为例，最小实现：
```python
from mcp.server import Server, stdio_server
from mcp.types import Tool, TextContent

server = Server("weather-server")

@server.list_tools()
async def list_tools() -> list[Tool]:
    return [Tool(name="get_weather", description="获取指定城市的天气",
                 inputSchema={"type":"object","properties":{"city":{"type":"string"}},"required":["city"]})]

@server.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name == "get_weather":
        city = arguments["city"]
        # ...实际调用API
        return [TextContent(type="text", text=f"{city}：晴，22°C")]
    raise ValueError(f"Unknown tool: {name}")

# 启动 stdio 传输
stdio_server.run(server)
```
配置文件（如 Claude Desktop）只需指向该脚本，Host 便可自动发现并使用该工具。

## 五、回归智能体架构：插件体系的位置

在 Day1 的五组件架构中，**插件体系是“工具”模块的具体实现形态**，并深度参与了规划与执行：

- **规划模块**：依赖工具/Skill 的元描述来制定计划。规范的描述能显著提升任务分解的准确性。
- **执行模块**：通过 MCP 等标准化协议，统一调用远程/本地工具，屏蔽异构性。
- **反思模块**：利用工具返回的结构化错误与结果，判断行动是否成功、是否需修正计划。

至此，智能体从纯粹的内在思维（Day2 推理范式），到意识自身的记忆局限（Day3 Token 边界）并设法突破（Day4 上下文管理），终于获得了**安全、标准化、可无限扩展的外部行动力**。Skill 和 MCP 就是它触及真实世界的双手。
